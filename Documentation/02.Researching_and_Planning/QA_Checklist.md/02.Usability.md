# Usability

### UI/UX Formatting

|What Does This Mean For The Project?|How and When Will This be Checked?|
|---|---|
|The design of the UI must be effective and efficient for a user to utilise|Will send templates of the UI to the client to check if the UI is effective for someone completely new and in flight. This will occur before the UI is designed and any changes required by the client will result in a new diagram of the UI. This will be repeated until the UI is approved by the client and is deemed achievable and realistic for me to code.|
|The design must be large enough and easy to use whilst in the air. This means that the sizing should be pretty large such that the user doesnt need to pay too much attention to the screen and visible from a distance.|This can once again be checked in flight or on the ground by the client before the designing stage and after as well according to standards set by the client.|
|The size should be adjustable according to the device used as it may be a mobile phone, tablet/iPad, or a laptop.|This can be tested through the use of various devices after coding the UI.|
|The UI should be designed such that it is easy to navigate (intuitive) such that it doesn't distract the pilot mid-flight. This will require rigorous testing and back-and-forth with the client.|This can be tested by the client and myself after the creation of the pages. Any required changes stated by the client will be done until the UI is staisfactorily navigatable.|

### Accessibility Considerations

|What Does This Mean for The Project?|How and When Will This be Checked?|
|---|---|
|This means that any styling done must be done such that the web app is usable by many people easily and especially must be done in a way such that in the air, it isn't very distracting allowing maximum focus on flying the plane. This includes having readable, and relatively large font sizes, have proper colour schemes, etc.|This will be tested by providing a sample of the UI in question to the client who will be asked to provide feedback or notes on what else may be required and these changes will be made.|
|There must be an effective use of space on the screen as the pilot may not be able to continuously scroll up and down for the required information. This means that it would be best to research the type of device used commonly among aviators and make the best fit for this device in specific. However, an adjustable format will also be utilised which changes depending on the size of the device used|This can be tested through the use of multiple devices to ensure that the UI is acceptable on all forms and can be handed to the client to review as well. This will most likely be tested after the UI is completed.|

### Usability and Accessibility of Buttons, Inputs, and Menus

|What Does This Mean for The Project?|How and When Will This be Checked?|
|---|---|
|The use of any buttons, menus, and inputs such as those for the airport, various functions, etc should be easily visible and labelled to ensure the proper understanding of each of these allowing the user to easily navigate through the app especially when in flight without much distraction|This can be checked through testing of sizes and going to the client and asking if the arrangement works well. This will most likely be done in the planning stage of the design with the aid of a diagram showing what the app will look like to allow for a visualisation of the process.|
|Responsiveness will be key to speed up the process as this is made to help save time for the pilot. This means that the inputs should be restricted to what is necessary for each flight for example and implementing strategies like lazy loading to ensure that only what is necessary is being processed.|This can be tested through testing all cases and ensuring that the implementation of all the features doesn't affect the processing of the inputs. This will most likely be tested after key functions are coded as well as after the program is completely finished.|
