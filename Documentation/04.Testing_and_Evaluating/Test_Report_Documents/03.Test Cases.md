# 3. Test Cases

**Test Case ID:** TC-01

<br>

**Feature:** Route Planner, Wind Preview panel

<br>

**Test Steps:** 1. Open Route Planner and build a route. 2. Open the Wind Preview panel. 3. Look for any way to pull live wind from Open-Meteo (as the Go/No-Go page does). 4. Enter TAS/wind/magnetic variation manually and check the solved heading/groundspeed.

<br>

**Expected Result:** The wind-triangle maths (heading, WCA, groundspeed) should be correct for manually-entered values, and ideally a live-weather option should be available like on the Go/No-Go page.

<br>

**Actual Result:** The manual wind-triangle calculation itself is correct (self-test panel, all cases pass). However, there is no live-weather option at all on this panel. It is manual-entry only, unlike Go/No-Go which calls Open-Meteo. This matches Bug #1.

<br>

**Pass/Fail:** Partial (calculation logic passes, missing live-weather integration is logged as Bug #1)

**Test Case ID:** TC-02

<br>

**Feature:** Route Planner and NAVLOG, departure time field

<br>

**Test Steps:** 1. Open the Time and ETA preview on Route Planner (and separately the NAVLOG page). 2. Click the departure time field. 3. Observe the browser's native time picker. 4. Enter a time and check the field-error text.

<br>

**Expected Result:** Since the field's error message says "Enter a time as HH:MM, 24-hour", the picker and the stored value should both behave as 24-hour to avoid confusing the pilot.

<br>

**Actual Result:** The stored value (`input.value`) is always a valid 24-hour `HH:MM` string and `validateDepartureTime()` correctly enforces this with its regex, so the calculation is never actually wrong. However, on a browser/OS locale set to 12-hour time, the native `<input type="time">` picker visually displays and scrolls in 12-hour AM/PM format while the help text says "24-hour", which is confusing for a pilot quickly glancing at the field. This matches Bug #2.

<br>

**Pass/Fail:** Partial (underlying logic passes, display/label mismatch is logged as Bug #2)

<br>

**Test Case ID:** TC-03

<br>

**Feature:** Logbook, logged-in state detection

<br>

**Test Steps:** 1. Log in with a test account. 2. Open the Logbook page. 3. Observe whether the "log in to view your logbook" prompt appears. 4. If it appears, click its "log in" button and observe the result.

<br>

**Expected Result:** Since the user is already logged in, the Logbook should show the flight list directly, not the logged-out prompt, and if the button were ever shown, it should open the login dialog, never sign the user out.

<br>

**Actual Result:** The logged-out prompt can incorrectly appear while still logged in (a timing/race issue between `logbook.js`'s own `supabase.auth.getUser()` check and the session state `auth.js` already holds). Clicking its button calls `document.getElementById("account-btn").click()`, and since `auth.js`'s `accountBtn` handler signs out whenever it believes a user is currently logged in, this logs the user out instead of opening the login dialog. This matches Bug #3.

<br>

**Pass/Fail:** Fail, logged as Bug #3

<br>

**Test Case ID:** TC-04

<br>

**Feature:** Go/No-Go, decision logic (unit and system test)

<br>

**Test Steps:** 1. Expand the Go/No-Go Self-test panel and confirm all cases pass. 2. Manually enter minimums (max wind 20kt, max crosswind 12kt, min visibility 8km, max cloud 70 percent) and conditions comfortably inside them. 3. Repeat with conditions just over one limit. 4. Repeat with flight condition set to "night" without night-currency ticked.

<br>

**Expected Result:** Self-test panel passes 8/8. Comfortable conditions give "Go". Conditions just over a limit give "No-Go" with a clear reason. Night flight without currency gives an automatic "No-Go" regardless of weather.

<br>

**Actual Result:** All three manual scenarios matched the expected verdicts, and the self-test panel showed 8 / 8 passing.

<br>

**Pass/Fail:** Pass

**Test Case ID:** TC-05

<br>

**Feature:** Security, cross-account data access (Row Level Security)

<br>

**Test Steps:** 1. Create two test pilot accounts (A and B) and save a NAVLOG to the logbook under Account A. 2. Log out and log in as Account B. 3. Open the Logbook. 4. In the browser console, attempt `supabase.from("flights").select("*")` as Account B.

<br>

**Expected Result:** Account B should never see Account A's flights, either through the UI or a direct client-side query, because Row Level Security is scoped to `auth.uid() = pilot_id`.

<br>

**Actual Result:** Account B's Logbook shows only Account B's own flights (or none), and the direct console query returns only rows owned by Account B. Account A's data was not visible or reachable from Account B's session.

<br>

**Pass/Fail:** Pass
