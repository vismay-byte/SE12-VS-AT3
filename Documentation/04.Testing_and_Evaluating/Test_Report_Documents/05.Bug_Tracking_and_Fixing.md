# 5. Bug Tracking & Fixing

**Bug ID:** BUG-01

<br>

**Description:** The Route Planner's Wind Preview panel is manual-entry only (TAS, wind direction/speed, magnetic variation). It is not connected to the Open-Meteo weather API, unlike the Go/No-Go page which does fetch live conditions. A pilot using the Wind Preview to sanity-check headings before building the full NAVLOG has no live-weather shortcut there and has to re-key numbers already fetched on the Go/No-Go page.

<br>

**Severity: Medium**

<br>

**Status: Open**

<br>

**Fix Details:** Reuse the existing Open-Meteo fetch already written for `js/gonogo.js` (same request shape: latitude/longitude of the departure aerodrome) and add a "Use live weather" control to the Wind Preview panel in `js/route-planner-bundle.js` that populates the wind direction/speed fields, leaving manual entry available as a fallback (e.g. offline, or for training on a hypothetical forecast wind).

<br>

<br>

**Bug ID:** BUG-02

<br>

**Description:** The departure-time fields (Route Planner's Time & ETA preview, and NAVLOG) use a native `<input type="time">`. Its stored value is always a correct 24-hour `HH:MM` string and is validated as such by `validateDepartureTime()`, but the browser's own picker UI can display and scroll in 12-hour AM/PM format depending on the device/OS locale, while the field's helper and error text explicitly say "24-hour". This is confusing for a pilot quickly entering a time, even though no incorrect time is ever actually calculated.

<br>

**Severity: Low**

<br>

**Status: Open**

<br>

**Fix Details:** Add a short inline note near the field along the lines of "shown in your device's format, saved as 24-hour" so the helper text stops implying the picker itself must be typed as 24-hour. A stretch option is a custom two-part HH / MM numeric input instead of the native time picker, which would guarantee a consistent 24-hour display across every browser/OS, at the cost of losing the native picker's built-in accessibility and mobile keyboard behaviour.

<br>

<br>

**Bug ID:** BUG-03

<br>

**Description:** The Logbook page can show its "log in to view your logbook" prompt even while the pilot is already logged in, and clicking that prompt's button signs the user out instead of opening the login dialog. Root cause: `js/logbook.js` keeps its own `currentUser` (from a fresh `supabase.auth.getUser()` call) separately from `js/auth.js`'s `currentUser` (kept from `onAuthStateChange`/`getSession()`), so the two can briefly disagree. The logged-out prompt's button (`loggedOutBtn`) does not open the auth dialog itself, it just forwards its click to the header's account button, which signs out whenever *its own* `currentUser` is truthy. If `logbook.js` briefly (or incorrectly) shows the logged-out state while `auth.js` still correctly believes the pilot is logged in, that forwarded click signs the pilot out.

<br>

**Severity: High**

<br>

**Status: Open**

<br>

**Fix Details:** Two changes needed. First, have the logged-out prompt's button call `openDialog("login")`-equivalent behaviour directly (e.g. export and call the dialog-opening function from `js/auth.js`) instead of forwarding a click to the account button, so it can never accidentally trigger a sign-out. Second, have `js/logbook.js` read the same authentication state `js/auth.js` already tracks (e.g. via the shared `onAuthStateChange` callback, or a small shared auth-state module) instead of independently re-querying `supabase.auth.getUser()`, removing the two-source-of-truth race condition entirely.

<br>
