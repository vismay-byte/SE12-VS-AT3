# 1. Overview of the Test Plan

**1.1 Purpose**

The purpose of this test plan is to ensure that the software project functions correctly, meets the specified requirements, and is free of major defects before deployment.

<br>

**1.2 Scope**

This test plan covers the following features:

- User authentication (login and registration)
- Form validation and error handling
- Database interactions
- Performance and security checks 

<br>

**1.3 Project Description**

The VFR NAVLOG Trainer is a browser-based Progressive Web App (PWA) for student pilots (RPL/PPL) training around Bankstown, Camden and the Sydney Basin. It is a training/demonstration tool, not a certified navigation product. Its key functionalities are: a Route Planner (waypoints, leg distance/true course); NAVLOG generation (wind-triangle heading/groundspeed, ETE/ETA, fuel required, PDF export); a Nav Page (aerodrome details, nearby aerodromes/waypoints); a Go/No-Go check (personal minimums vs. live Open-Meteo weather); Checklists (per-phase, per-aircraft); and a Logbook (past flights and running hours, saved to the pilot's account).

Technologies used: HTML5, CSS3 and vanilla JavaScript (ES modules, no framework/build step); a Service Worker for offline app-shell caching and installability; Supabase (PostgreSQL + Auth) for accounts and the logbook, with Row Level Security policies restricting every pilot to their own data; the Open-Meteo API for live weather; and jsPDF (loaded on demand) for NAVLOG PDF export.

<br>

**1.4 Testing Objectives**

- Verify that all features work as expected

- Identify and fix bugs before deployment

- Performance, usability, and security standards

# 2. Testing Strategy

**2.1 Types of Testing**

*Unit Testing:*

Testing individual functions in isolation, with known inputs and hand-checkable expected outputs. This project's calculation modules (`solveLeg()` in `js/route-planner-bundle.js`, `evaluateGoNoGo()` in `js/gonogo.js`, `crosswindComponentKt()`, `fuelRequired()`, `formatHoursDecimal()` in `js/logbook.js`, etc.) each ship their own `SELF_TEST_CASES` array and `runSelfTests()` function, rendered as a collapsible "Self-test" panel on that feature's page (via `renderSelfTestList()` in `js/shared-utils.js`). How I do this: open each feature page, expand its Self-test panel, and confirm the pass count reads e.g. "8 / 8 self-test cases pass"; if any case fails, the panel shows the expected vs. actual value for that case so I can find the fault without a debugger. I also run `node --input-type=module --check` over every file in `js/` as a fast syntax-level unit check before testing behaviour.

*Integration Testing:*

Checking that modules which depend on each other actually agree, not just that each works alone. For example: does the Route Planner's saved draft (`sessionStorage`, key `navlog-route-draft-v1`) actually get read correctly by the NAVLOG page and by the Nav Page's "nearby" lookups? Does a NAVLOG built from a route actually save a `flights` row that the Logbook can read back? I test this by performing the real end-to-end action (build a route → open NAVLOG → save to logbook → open Logbook) rather than testing each page in isolation.

*System Testing:*

Testing the whole app as a real user would, end-to-end, in a real browser rather than by calling functions directly. This means: loading `index.html` from a local static server (`python -m http.server 8080`, since service workers refuse to register over `file://`), clicking through every feature card from the home screen, and confirming the full flow in the main flowchart works (home screen → feature → input → validation → calculation → output → back to home). System testing is where the four bugs logged in `05.Bug_Tracking_and_Fixing.md` were actually found, because each is only visible once features are used together in a real session (e.g. being logged in already, then opening the Logbook).

*User Acceptance Testing (UAT):*

Sending the built app (or screenshots/short recordings of it) to my client and asking them to try the Route Planner → NAVLOG → Checklists flow as if prepping for a real training flight, then noting anything confusing or wrong against real flight-training conventions (e.g. is the fuel reserve rule realistic for day vs. night VFR?).

*Performance Testing:*

Checking page load speed, responsiveness and unnecessary work using Chrome DevTools rather than guesswork. Concretely: DevTools → **Lighthouse** panel, run a Performance audit on the deployed (GitHub Pages/local server) site for both Mobile and Desktop presets, and note the score plus any flagged "Reduce unused JavaScript"/"Serve images in next-gen formats" items; DevTools → **Network** tab (with cache disabled once, then enabled) to check that repeat visits are served from the Service Worker cache almost instantly, and that the Open-Meteo/jsPDF/Supabase calls only fire when a feature that actually needs them is used (lazy-loading), not on every page load; DevTools → **Performance** tab to record a trace while filling in a full NAVLOG and check there's no long layout/scripting jank on input.

*Security Testing:*

Ensuring protection against vulnerabilities like Cross-Site Scripting (XSS), broken access control and weak authentication, using both manual review and an automated scanner. Concretely: run **OWASP ZAP**'s automated scan against the locally-served app (`http://localhost:8080`) to catch missing security headers and any obvious injection points; manually type XSS payloads (e.g. `<img src=x onerror=alert(1)>`) into every text field that gets rendered back to the screen (custom waypoint names, display name, aircraft type) and confirm they show as literal text, not run as a script, because of `escapeHtml()`; and in the Supabase dashboard, log in as two different test accounts and confirm one account can never see or delete another's `flights`/`pilots` rows (Row Level Security), which is the real access-control boundary in this app rather than anything enforced only in the browser. There is no SQL Injection surface to test directly, since every database call goes through the Supabase client library (`supabase.from(...).select()/.eq()`), which parameterises values rather than building raw SQL strings.

Chrome, Firefox, Safari (incl. iOS Safari, since it never fires `beforeinstallprompt` — see Bug #4), Edge

Android phone, iPad/tablet, Windows laptop, using Chrome DevTools' device-emulation toolbar for extra screen sizes I don't have physical access to
