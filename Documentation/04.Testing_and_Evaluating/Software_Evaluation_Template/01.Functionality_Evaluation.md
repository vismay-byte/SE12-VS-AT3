### Achievement of Requirements and Objectives

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|Ensure that the features provided actually help the pilot stay on time and on track rather than distract.|Yes|The light/dark theme switch lets a student see everything clearly with no visual distraction. The audio callout system is adjustable to the student's own settings (timing and volume), so callouts are anticipated rather than a surprise, and the checklists use simple dropdowns and large tap targets so most of the list stays visible from a distance without needing to "tick off" every item mid-flight. A printable PDF export is also available for students who would rather not look at a screen at all in the air.|
|Make sure that the calculations are correct so the pilot doesn't enter the wrong amount of fuel or miss a checklist item due to an error in the code.|Yes|Significant research went into the underlying aviation maths (wind triangle, fuel/reserve rules) and my client helped fine-tune a few misunderstandings from that research. Each calculation module also ships its own hand-checkable Self-test panel, so the expected value for a given input is worked out by hand rather than copied from a table, which builds trust that the maths matches real navigation procedure.|
|Have correct information for each aerodrome and ensure it is properly up to date.|Yes|Aerodrome data was researched from official CASA documents and cross-checked against sites like FlightRadar24 for scale and layout. A free, no-signup, no-API-key weather source (Open-Meteo) was also found and integrated for the Go/No-Go check, though this only reflects real-time conditions rather than a forecast ahead of time.|

### Feature Function

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|Ensure that all calculations are done properly.|Yes|Every calculation module (wind triangle, time/ETE/ETA, fuel and reserve, Go/No-Go) has its own Self-test panel comparing computed output against hand-worked expected values across a range of scenarios, rather than relying on manual spot-checks alone.|
|Ensure that there are no bugs within the actual program.|Partially|The core calculation logic has no known bugs and all Self-tests pass. However, going through the app feature by feature during testing did surface four real issues (documented in 05.Bug_Tracking_and_Fixing.md), covering the Route Planner's Wind Preview panel, the departure time field's display format, a Logbook login state bug, and the Install app button. None of these affect the correctness of the core calculations themselves.|

### Forms and Input Validations and Incorrect Input Handling

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|Ensure that all data that is inputted is sanitised to ensure security.|Yes|Every string rendered back to the screen (waypoint names, aircraft type, saved flight data) is passed through a shared `escapeHtml()` function before being written into the DOM, so characters used in script-injection attempts are shown as harmless text rather than executed. This was tested by entering payloads such as `<img src=x onerror=alert(1)>` into text fields and confirming they render as plain text.|
|Ensure that any information that needs to be entered is in the correct format and structure.|Yes|Numeric fields (true course, wind speed/direction, TAS, magnetic variation) are checked against a strict min/max range with a shared `numericFieldCheck()` helper, and the departure time field is checked against a strict 24-hour `HH:MM` regular expression. A visible, specific error message appears next to the field whenever a value fails validation, rather than the input silently being ignored or "cleaned".|

<br>

### APIs and Third-Party Integration Function

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|APIs are used to retrieve information regarding airport weather, conditions and procedures.|Yes|Open-Meteo is called from the Go/No-Go page to compare live wind, visibility and cloud cover against the pilot's personal minimums. This was checked against the BOM website and other weather apps during development to confirm accuracy for the same time and location.|
|APIs may be utilised within this project for geolocational purposes.|Partially|Aerodrome and waypoint coordinates are stored as static data rather than pulled from a live geolocation API, and nearby-aerodrome/waypoint lookups on the Nav Page use this static data rather than the device's own GPS position. This was a deliberate scope decision given the fixed, small set of Sydney Basin aerodromes covered, rather than an unfinished feature.|
