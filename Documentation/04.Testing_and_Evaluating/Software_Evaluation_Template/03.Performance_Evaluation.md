# Performance

---

### Optimised Loading Times

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|Code should be minimal for what is actually running at the time.|Partially|Lazy loading is implemented properly. `js/shell.js` only imports a feature's own JavaScript module the first time that view is opened, rather than loading every feature's code on the home screen, and the jsPDF/supabase-js CDN imports only load when a page actually needs them. However, there is no build step (no minification or bundling), so CSS/JS ship as readable, commented source rather than a production-sized bundle. This was a deliberate trade-off to keep the code easy to mark and debug directly, at the cost of larger raw file sizes.|
|There should be no unnecessary code occupying space when the program runs.|Yes|A "Session 15" refactor merged what were previously 12 separate per-feature stylesheets and several separate JS files into shared bundles for files that always load together on the same page, reducing the number of separate files and network requests without changing behaviour.|

<br>

### Optimising Database Queries

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|Only the necessary data should be selected from the database per user.|Yes|`js/logbook.js`'s `loadFlights()` selects only the specific columns the UI needs and filters with `.eq("pilot_id", currentUser.id)`, so it never requests another pilot's rows client-side, on top of Row Level Security enforcing the same boundary at the database.|
|Caching should be used to avoid repeating the same data request.|Yes|Both `pilots` and `flights` use `uuid` primary keys, which Postgres indexes automatically, and every lookup performed (`auth.uid() = pilot_id`, `auth.uid() = id`) is an indexed-key comparison rather than a full-table scan. No custom indexes were needed given the project's scale (a training app for a small group of student pilots).|

<br>

### Unnecessary Code and API Calls Removed

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|The code should be cleansed regularly so anything unused or overly complex is simplified.|Yes|API calls are one-off and on-demand rather than polling. The Open-Meteo fetch in `js/gonogo.js` only runs when the pilot clicks to fetch live weather, and jsPDF is only requested when "Export as PDF" is actually clicked, never preloaded.|
|Caching should be used so data that doesn't need to be re-requested isn't.|Yes|`js/route-planner-bundle.js` caches both the aerodrome list (`aerodromesByIdCache`) and the aircraft list (`aircraftListCache`) the first time each is fetched, so moving between the Wind Preview, Time and ETA, and Fuel panels on the same page does not re-fetch data already in memory.|

<br>

### Utilise Caching Effectively

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|Necessary data (such as user data) should be retrieved only once.|Yes|`service-worker.js` implements a cache-first strategy for every same-origin static file (HTML, CSS, JS, manifest, icon, and the aerodrome/waypoint/aircraft JSON data), so a returning pilot's app shell and static data load from the cache instead of the network, and the shell still works offline. The `activate` handler also clears any old cache version when `CACHE_NAME` is bumped, so stale caches don't build up. Live, per-request data (Open-Meteo weather, Supabase auth/logbook data, and the on-demand jsPDF/supabase-js CDN imports) is deliberately excluded from this cache, since caching those could show a pilot stale weather or an out-of-date logbook.|
