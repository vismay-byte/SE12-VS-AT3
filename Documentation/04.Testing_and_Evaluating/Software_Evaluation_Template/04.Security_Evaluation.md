# Security

---

### User Authentication and Authorisation

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|Student private data (past trips, personal info) is secured and can only be accessed by that student.|Yes|Authentication is handled by Supabase Auth (`js/auth.js`), which issues a signed session token rather than anything home-grown. Authorisation is enforced by PostgreSQL Row Level Security in `supabase/schema.sql`, with every policy on `pilots` and `flights` scoped to `auth.uid()`, so it holds at the database level even if a request bypassed the app's own UI. This was verified in system testing using two separate test accounts (TC-06 in `03.Test Cases.md`).|
|No user should be able to alter the code through inputs, requiring data sanitisation for input fields.|Yes|See the Validation and Sanitisation table below, this is covered there in detail since it is the same requirement.|

<br>

### Validation and Sanitisation

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|A sanitisation function is needed for all major input fields so malicious scripts cannot affect the app.|Yes|Every string from user input or the Supabase database (waypoint names, aircraft type, saved flight dates/routes) is passed through a shared `escapeHtml()` function before being written into the DOM, converting characters like `<`, `>`, `"`, `'` and `&` into inert HTML entities. This was manually tested with payloads such as `<img src=x onerror=alert(1)>` in the custom waypoint name and signup display name fields, both rendered as literal text rather than executing.|
|A validation function is needed for each input region so only valid data types/ranges are entered.|Yes|Every numeric field is checked against a strict min/max range via `numericFieldCheck()`, and the departure time field is checked against a strict 24-hour `HH:MM` regular expression, before the value is used in any calculation.|

<br>

### Password Hashing

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|Login and user details should be stored securely, ideally via a backend rather than the project's own code.|Yes|Password hashing is delegated entirely to Supabase Auth. `js/auth.js` never stores or transmits a plain-text password beyond the initial HTTPS request, and this was checked in the Supabase dashboard's Table Editor, where `auth.users` stores only a hashed credential. `js/auth.js` also enforces a client-side password policy (length, uppercase, lowercase, digit, symbol) so a pilot gets instant feedback, and surfaces the server's own rejection reason if Supabase's policy is stricter than the local check.|

<br>

### Enforcing HTTPS

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|HTTPS should be enforced if the project is deployed online.|Yes|GitHub Pages (the intended host) serves every page over HTTPS automatically and cannot be reconfigured to serve plain HTTP. Every external call the app makes (Supabase, Open-Meteo, jsPDF) is already an `https://` URL, and service workers refuse to register at all over plain `http://` except on `localhost`, so HTTPS is effectively required for the app to function as intended, not just enforced as a setting.|
