# Testing

---

### Unit Testing

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|Functions or components are tested individually once completed, to ensure they work within themselves.|Yes|Every calculation module (`solveLeg()` in `js/route-planner-bundle.js`, `evaluateGoNoGo()` in `js/gonogo.js`, `crosswindComponentKt()`, `fuelRequired()`, `formatHoursDecimal()` in `js/logbook.js`, and others) ships its own `SELF_TEST_CASES` array and `runSelfTests()` function, shown as a collapsible Self-test panel on that feature's page. Each case's expected value is worked out by hand rather than copied from a table, and the panel reports a pass count plus per-field differences for any failure.|

<br>

### Integrated Testing

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|Different functions and components are combined and tested together to ensure they work well in unison.|Yes|Real end-to-end actions were tested rather than only isolated functions, for example building a route, then opening NAVLOG, then saving to the logbook, then opening the Logbook, to confirm the route draft saved in `sessionStorage` is read correctly by later pages and that a NAVLOG actually produces a `flights` row the Logbook can read back.|

<br>

### User Testing for Feedback

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|The client (and potentially other student pilots or instructors) should be asked to test the program and give feedback on its usefulness in flight or on the ground.|Partially|The app was shared with my client (a student pilot) throughout development for feedback on individual pages and wireframes, but a full end-to-end user acceptance pass with the completed app, and feedback from additional student pilots or instructors beyond my one client, is still to be arranged now that the build itself is finished.|

<br>

### Resolved and Documented Bugs

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|Issues identified during testing, and what was done to fix them, should be documented for future debugging.|Partially|Four real issues found while testing the finished app against the QA Checklist are documented in `05.Bug_Tracking_and_Fixing.md` with a description, severity, status and fix plan for each. All four are currently logged as Open rather than Closed, since the write-up and fix work happened in the same session, so the fixes themselves are the next step rather than already applied and verified.|
