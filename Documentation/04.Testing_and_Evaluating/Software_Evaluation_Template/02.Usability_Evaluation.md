# Usability

---

### UI/UX Formatting

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|The design of the UI must be effective and efficient for a user to utilise.|Yes|The home screen is a simple layout of six feature cards, a login box and a logo, with self-explanatory labelled buttons throughout. Each feature's layout mirrors the order a student pilot would normally read the same information from paper notes, so fields can be filled in without jumping between documents.|
|The design must be large enough and easy to use whilst in the air, visible from a distance.|Yes|The default scaling is based on a tablet, the most common device for student pilots, which loads quickly and keeps spacing wide enough to read from a distance.|
|The size should be adjustable according to the device used (mobile, tablet/iPad, or laptop).|Yes|Adjustable rates and set ratios are used across the CSS so different devices share a consistent scale rather than needing separate layouts per device.|
|The UI should be designed so it is easy to navigate (intuitive) and doesn't distract the pilot mid-flight.|Yes|Feature cards are ordered left to right, top to bottom, following the order a pilot would normally complete the same tasks in, and every field and button is clearly labelled so intent is obvious without extra explanation.|

<br>

### Forms Validation Messages

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|Each input given by the user must be validated depending on the conditions of that field (aerodrome, flight time, etc).|Yes|Toast notifications confirm actions such as loading a previous route or resetting information, and any invalid input produces a clear, specific error message next to the field rather than a generic warning.|

<br>

### Accessibility Considerations

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|Styling must make the app usable by many people easily, and not distracting in the air.|Yes|Font sizes are kept relatively large and readable from a distance, with colour schemes chosen for clarity rather than decoration.|
|Effective use of screen space is needed, since the pilot may not be able to continuously scroll.|Yes|Checklists in particular use dropdowns and a simple tap-to-tick interaction so most, if not all, of a checklist phase stays visible on screen at once, minimising scrolling.|
|Colours used should be effective for pilots in any lighting condition.|Yes|A light/dark theme switch is implemented, covering both daylight ground use and low-light cockpit use, rather than a single fixed colour scheme.|

<br>

### Usability and Accessibility of Buttons, Inputs, and Menus

|Criteria|Has the Criteria Been Achieved?|Analysis|
|---|---|---|
|Buttons, menus and inputs should be easily visible and labelled so their purpose is understood without distraction.|Yes|Every feature card has a plain-English title and a one-line description of what it does. Buttons describe the action directly (for example "Save to logbook", "Export as PDF", "Re-export PDF"), and fields with a less obvious format (departure time, magnetic variation) show an example or accepted range alongside the label.|
|Responsiveness should be prioritised, restricting inputs to what's necessary and using strategies like lazy loading.|Yes|Inputs are kept to only what a given calculation needs (the Wind Preview only asks for TAS, wind direction/speed and magnetic variation, not the whole NAVLOG's fields). `js/shell.js` also lazy-loads each view's own module only when that view is opened, and the service worker caches the app shell, so repeat visits to any page are close to instant.|
